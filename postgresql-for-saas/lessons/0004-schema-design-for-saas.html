Schema Design for SaaS

Modeling users, organizations, memberships, and subscriptions — a complete worked example

⏱ ~20 minutes 📖 Read + quiz 🎯 Practical design

This lesson builds the schema you'd actually ship for a multi-tenant SaaS product. We'll go from "what entities exist?" to a complete, production-ready CREATE TABLE set — step by step.

Step 1: Identify your entities

Every SaaS product manages roughly the same core concepts:

  • Users — people with accounts
  • Organizations — teams or companies (the billing unit)
  • Memberships — which users belong to which orgs, and in what role
  • Plans — the tiers you offer (free, pro, enterprise)
  • Subscriptions — which org is on which plan, and since when

Each entity becomes a table. The relationships between them become foreign keys and junction tables.

Step 2: Map the relationships

users ↔ organizations many-to-many (via org_memberships)
organizations → subscriptions one-to-many (an org has one active subscription at a time)
subscriptions → plans many-to-one (many subscriptions share a plan)

The trickiest relationship is users ↔ organizations. A user can belong to multiple orgs (invited to a work org and a personal one). An org can have many users. This is a many-to-many relationship, and it requires a junction table to model correctly.

Why a junction table?

You can't store "Alice belongs to Acme and Globex" in the users table without repeating data. You can't store "Acme has Alice and Bob" in the organizations table cleanly either. A separate org_memberships table with (user_id, org_id, role) rows is the standard solution — one row per relationship, no repetition.

Step 3: Write the schema

Users

CREATE TABLE users (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  email      TEXT        NOT NULL UNIQUE,
  name       TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

Note the choices: UUID primary key (globally unique, safe to expose in URLs), TIMESTAMPTZ (stores timezone info — always use this over plain TIMESTAMP), and UNIQUE on email (enforced at the database layer).

Organizations

CREATE TABLE organizations (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  name       TEXT        NOT NULL,
  slug       TEXT        NOT NULL UNIQUE, -- for URLs: acme.yourapp.com
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

Org memberships (the junction table)

CREATE TABLE org_memberships (
  user_id    UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  org_id     UUID        NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  role       TEXT        NOT NULL DEFAULT 'member'
                          CHECK (role IN ('owner', 'admin', 'member')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, org_id)
);

Key details here:

  • ON DELETE CASCADE — if a user is deleted, their memberships are automatically deleted too. No orphaned rows.
  • CHECK (role IN (...)) — the database rejects any role value not in this list. No "superadmin" typos slipping through.
  • PRIMARY KEY (user_id, org_id) — a composite primary key that prevents duplicate memberships (a user can only be in an org once).

Plans

CREATE TABLE plans (
  id                 UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
  name               TEXT    NOT NULL UNIQUE, -- 'free', 'pro', 'enterprise'
  monthly_price_cents INTEGER NOT NULL CHECK (monthly_price_cents >= 0)
);

Currency is stored as cents (an integer), not dollars (a float). Floating-point arithmetic is not safe for money — 0.1 + 0.2 is not exactly 0.3 in IEEE 754. Integers are exact. Divide by 100 at display time.

Subscriptions

CREATE TABLE subscriptions (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id     UUID        NOT NULL REFERENCES organizations(id),
  plan_id    UUID        NOT NULL REFERENCES plans(id),
  status     TEXT        NOT NULL DEFAULT 'active'
                          CHECK (status IN ('active', 'past_due', 'canceled')),
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at TIMESTAMPTZ  -- NULL means open-ended
);

Step 4: Understand what normalization gave you

Notice what's not in the schema: the plan name and price don't appear in subscriptions. You might be tempted to copy plan_name into every subscription row so you don't need a JOIN to look it up. This is called denormalization, and it creates a problem: when you rename "Pro" to "Pro Plus", you'd have to update thousands of subscription rows.

Normalization means each fact exists in exactly one place. The plan name lives in plans. Subscriptions reference it. You JOIN when you need it. One source of truth, one place to update.

Normalized (what we built)Denormalized (what to avoid)
Plan name stored inplans.name — oncesubscriptions.plan_name — every row
Update plan nameOne UPDATE to plansUPDATE thousands of subscription rows
Data consistencyGuaranteed by FKApplication must keep in sync
Query complexityNeed a JOINNo JOIN needed

Step 5: A real query against your schema

"Show me all active subscribers and what plan they're on:"

SELECT
  organizations.name  AS org_name,
  plans.name          AS plan_name,
  subscriptions.started_at
FROM subscriptions
JOIN organizations ON subscriptions.org_id  = organizations.id
JOIN plans         ON subscriptions.plan_id = plans.id
WHERE subscriptions.status = 'active'
ORDER BY subscriptions.started_at DESC;

This returns a clean table of org names, plan names, and start dates — all in one query, because the relationships are modeled explicitly in the schema.

Retrieval practice — 3 questions

Resist the urge to scroll. The recall effort builds the memory.

1. The org_memberships table exists to model what kind of relationship?
Correct. Users can belong to many orgs, and orgs can have many users — that's many-to-many. A junction table (also called a join table or bridge table) with (user_id, org_id) rows is the standard pattern for modeling it.
Not quite. The user ↔ organization relationship is many-to-many — a user can be in many orgs, and an org can have many users. A junction table like org_memberships holds one row per (user, org) pair to represent this.
2. Normalization means each piece of data is:
Correct. Normalization means one fact, one place. The plan name lives in plans — not repeated in every subscription row. When it changes, you update one row, not thousands.
Not quite. Normalization is the opposite: each fact is stored in exactly one place. Copying data into multiple tables is called denormalization — it can improve query speed but creates consistency problems when that data changes.
3. Why is monthly_price_cents INTEGER preferred over monthly_price FLOAT for money?
Correct. Floating-point numbers (FLOAT, DOUBLE) cannot represent all decimal values exactly — 0.1 + 0.2 ≠ 0.3 in IEEE 754 arithmetic. For currency, you need exact arithmetic. Store as integer cents, divide by 100 at display time.
The key reason is correctness, not performance. Floating-point arithmetic has rounding errors — 0.1 + 0.2 in a float is not exactly 0.3. For money, that's dangerous. Storing cents as an integer gives you exact arithmetic: 10 cents + 20 cents = 30 cents, always.
