The Database Landscape

A map of your options — from what you already know to where PostgreSQL sits

⏱ ~15 minutes 📖 Read + quiz 🎯 Orientation

Starting where you are

You've used MongoDB or Firebase. You know what it feels like to store a JSON document in a collection and retrieve it by ID. That's a real foundation — not nothing.

But when you start building a SaaS product — something with user accounts, billing, relationships between entities, audit trails — you'll hear "use PostgreSQL" from a lot of experienced engineers. This lesson explains why, by first mapping the full landscape of database options so you understand the trade-offs rather than just following a recommendation on faith.

By the end of this lesson

You'll be able to name the five major families of databases, explain what makes each one distinctive, and articulate why SaaS products often end up choosing a relational database like PostgreSQL.

The two big families

Every database you're likely to encounter belongs to one of two broad families:

  • Relational databases (SQL) — data is stored in tables with rows and columns. A rigid schema defines the shape of data upfront. Records relate to each other via foreign keys. You query with SQL.
  • Non-relational databases (NoSQL) — a catch-all for everything else. Data might be stored as documents (JSON-like), key-value pairs, columns, or graphs. Schema is often flexible or nonexistent. Query languages vary.

"NoSQL" is an unfortunately vague term — it means "not relational," which doesn't tell you much. Within NoSQL there are at least four distinct sub-families with very different use cases.

Inside NoSQL: four sub-families

Since you've already used two of these, we'll use them as anchors.

1. Document stores  You've used these

MongoDB, Firebase/Firestore, CouchDB. Data is stored as JSON-like documents grouped into collections. Each document can have a different shape (schema-less). You fetch by ID or by querying fields within documents. The big trade-off: flexible and natural for hierarchical data, but querying across documents (especially when data is related) requires more planning, and joins are manual or nonexistent.

2. Key-value stores  Simplest model

Redis, DynamoDB (in simple mode), Memcached. Think of a giant dictionary: you look up a value by a key. No schema, no query language beyond get/set/delete. Blazing fast. Used for caching, sessions, rate limiting, leaderboards. Not a primary database for complex application data — you'd lose most of what makes databases useful.

3. Wide-column stores  Scale-first

Cassandra, Google Bigtable, HBase. Data is organized by rows, but each row can have a different set of columns. Designed for enormous write throughput at scale (think: millions of writes per second). Complex to model and operate. Almost never the right first choice for a new SaaS product.

4. Graph databases  Relationship-first

Neo4j, Amazon Neptune. Data is modeled as nodes and edges — people, connections between them, and the properties of those connections. Powerful for social graphs, recommendation engines, fraud detection. Niche — most SaaS applications don't need a dedicated graph database, though PostgreSQL can handle many graph-like queries through SQL.

Inside Relational: the main options

Now the family you're evaluating:

PostgreSQL  Our focus

Open-source, highly extensible, known for correctness and standards compliance. Supports JSON natively (JSONB), full-text search, geospatial queries, and hundreds of extensions. The de-facto standard for new SaaS products. Runs on every major cloud (AWS RDS, Google Cloud SQL, Supabase, Railway, Neon, etc.).

MySQL / MariaDB  Relational

Older and historically more widely deployed (the M in LAMP stack). Slightly simpler, somewhat less featureful than PostgreSQL. MariaDB is a community fork of MySQL. Most new projects choose PostgreSQL over MySQL; MySQL remains common in legacy systems and WordPress deployments.

SQLite  Embedded

A file-based relational database — the entire database is a single file on disk. No separate server process. Used in mobile apps, desktop apps, embedded systems, and as a local development database. Not designed for concurrent multi-user web applications; not what you'd run in production for a SaaS product.

Why PostgreSQL keeps showing up in SaaS

Think about what a typical SaaS application needs to manage:

  • Users — each with an account, profile, and settings
  • Organizations — users belong to orgs; orgs have billing plans
  • Subscriptions & invoices — financial data that must never be wrong or inconsistent
  • Feature usage & audit logs — who did what, when
  • The core domain — projects, documents, tasks, messages — whatever your app does

This data is deeply relational. A user belongs to an org, which has a subscription, which covers a set of features. If you store this in a document database, you're constantly working around the lack of joins — either embedding data everywhere (then keeping copies in sync) or writing complex application-level queries to stitch things together.

PostgreSQL gives you:

  • Structure you can trust. A schema means your database enforces that every row has the data your application expects — no surprises at runtime.
  • Joins. You can query across related tables in a single statement rather than making multiple round trips from application code.
  • ACID transactions. When you charge a card and update a subscription, either both happen or neither does. No half-complete states.
  • SQL. A 50-year-old standard understood by every BI tool, analytics platform, and data engineer on earth. Your data is not locked into a proprietary query language.
  • Flexibility where you need it. PostgreSQL's JSONB column type lets you store flexible, schema-less data inside a relational table. You get the benefits of both models.
The JSONB shortcut

One common misconception: "I need MongoDB because my data has varying shapes." PostgreSQL's JSONB type lets you store arbitrary JSON inside a column, and you can even index into it. You can have a users table with a structured email and created_at, and an unstructured metadata JSONB column for anything else. You often don't have to choose.

When PostgreSQL is not the right choice

It's worth being honest about the cases where another database makes more sense:

  • You need real-time sync to clients. Firebase/Firestore has built-in real-time subscriptions. PostgreSQL requires extra infrastructure (like Supabase Realtime or polling) to achieve this.
  • You're building a rapid prototype with highly uncertain schema. A document database lets you change your data shape freely without migrations. This advantage shrinks as the product matures.
  • You need to cache at extreme speed. Redis alongside PostgreSQL, not instead of it.
  • You're handling truly massive write throughput at IoT scale (millions of events/sec). Cassandra was built for this.

For most SaaS products, especially in their first several years, PostgreSQL is the default that you rarely regret. The ecosystem, tooling, and community are exceptional.

Retrieval practice — 3 questions

Answer from memory without scrolling up. Effort here builds long-term retention.

1. PostgreSQL belongs to which database family?
Correct. PostgreSQL is a relational database — data lives in tables with rows and columns, tied together with foreign keys. MongoDB and Firebase, which you've used, are document databases.
Not quite. PostgreSQL is a relational database — data lives in structured tables with enforced schemas. MongoDB and Firebase (which you've used) are the document databases.
2. MongoDB and Firebase/Firestore are examples of which NoSQL sub-family?
Correct. Both MongoDB and Firebase store data as JSON-like documents inside collections. That's the defining characteristic of document stores.
Not quite. MongoDB and Firebase are document stores — they store JSON-like documents in collections. Key-value stores (like Redis) are simpler: just a key that maps to a value, no document structure.
3. A SaaS app has users, billing plans, and invoices with complex relationships. Which database type is the strongest default?
Correct. Complex relationships (users → orgs → billing → invoices) are exactly what relational databases are designed for. SQL joins, ACID transactions, and enforced schemas make this kind of data reliable and queryable.
Think about the data: users belong to orgs, orgs have billing plans, billing plans have invoices. That web of relationships is the natural home of a relational database. Document databases make you handle joins in application code — workable, but increasingly painful as the data model grows.
