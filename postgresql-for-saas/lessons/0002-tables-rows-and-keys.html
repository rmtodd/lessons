Tables, Rows & Keys

How PostgreSQL organizes data — and why it's fundamentally different from MongoDB

⏱ ~15 minutes 📖 Read + quiz 🎯 Core concepts

From documents to tables

In MongoDB, a user lives in a collection as a JSON document. In PostgreSQL, that same user lives in a table as a row. The concepts map closely — but the implications are very different.

MongoDB — a user document
{
  "_id": "a1b2c3",
  "name": "Alice",
  "email": "alice@acme.com",
  "role": "admin"
}
PostgreSQL — a users table
SELECT * FROM users;

 id  │ name  │ email           │ role
─────┼───────┼─────────────────┼───────
 1   │ Alice │ alice@acme.com  │ admin
 2   │ Bob   │ bob@acme.com    │ member

The document and the row contain the same data. What changes is the structure enforcing that data.

Anatomy of a table

A PostgreSQL table is a grid:

  • Columns define the shape — each column has a name and a data type (TEXT, INTEGER, TIMESTAMPTZ, etc.). Every row in the table must conform to this shape.
  • Rows are the individual records — one row per user, one row per invoice, one row per event.
  • Schema is the full definition of what columns exist, what types they are, and what constraints they enforce (like "email must be unique" or "name cannot be null").

This is the core difference from MongoDB: you declare the shape of your data upfront, and the database enforces it on every insert and update. You can't accidentally store a user without an email if you've defined email TEXT NOT NULL.

Why this matters for SaaS

SaaS data tends to be business-critical — user accounts, billing records, access permissions. A document database lets you store a subscription record missing its plan_id. PostgreSQL rejects it at the database layer, not the application layer. Fewer bugs reach production.

Primary keys: the unique identifier

Every row needs a unique identifier so you can refer to it precisely. In MongoDB, that's _id — auto-generated and always present. In PostgreSQL, you declare a primary key column yourself.

CREATE TABLE users (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  email      TEXT        NOT NULL UNIQUE,
  name       TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

Breaking that down:

  • id UUID PRIMARY KEY DEFAULT gen_random_uuid() — a UUID column that PostgreSQL fills automatically, just like MongoDB's _id.
  • NOT NULL — this column can never be empty.
  • UNIQUE — no two rows can have the same email.
  • DEFAULT now() — automatically set to the current timestamp when a row is inserted.

Foreign keys: the link between tables

This is where relational databases earn their name — and where they diverge most from MongoDB.

Imagine you have an organizations table and want to track which org each user belongs to. In MongoDB, you might embed the org name inside the user document, or store an org_id field and manually ensure it matches a real org. PostgreSQL formalizes this with a foreign key:

CREATE TABLE org_memberships (
  user_id    UUID  NOT NULL REFERENCES users(id),
  org_id     UUID  NOT NULL REFERENCES organizations(id),
  role       TEXT  NOT NULL DEFAULT 'member',
  PRIMARY KEY (user_id, org_id)
);

REFERENCES users(id) tells PostgreSQL: every value in user_id must correspond to an existing row in users.id. If you try to insert a membership for a user that doesn't exist, the database rejects it. If you try to delete a user who has memberships, the database protects the relationship.

This is called referential integrity — the database guarantees your relationships are never broken, even under concurrent writes.

The MongoDB comparison

In MongoDB, if you store { user_id: "abc123", org_id: "xyz789" } in a memberships collection, nothing stops "abc123" from being a user that was deleted yesterday. You have to enforce this in application code. PostgreSQL does it at the database layer — always, for every write, from every connection.

Schema enforcement: the contract

The schema is a contract between your application and your database. PostgreSQL will hold you to it:

ConstraintWhat it enforcesExample
NOT NULLColumn cannot be emptyemail TEXT NOT NULL
UNIQUENo two rows can share a valueemail TEXT UNIQUE
PRIMARY KEYUnique + not null — identifies the rowid UUID PRIMARY KEY
REFERENCESValue must exist in another tableorg_id UUID REFERENCES orgs(id)
CHECKCustom rule must be trueCHECK (price_cents >= 0)
DEFAULTValue filled automaticallycreated_at DEFAULT now()

Every single write — from your API, your background jobs, your admin scripts, any database connection — is checked against all these constraints. There is no way to bypass them except by explicitly dropping them (which you would never do in production).

Retrieval practice — 3 questions

Answer from memory. Resist scrolling up — the effort is the point.

1. In a relational database, a single user record is called a:
Correct. In SQL databases, each record is a row. Collections of rows form a table. In MongoDB terms: a row ≈ a document, a table ≈ a collection — but with an enforced shape.
Not quite. In SQL, each record is a row inside a table. You may know this as a "document" from MongoDB — the concept is the same (one record), but in PostgreSQL it's called a row and must conform to the table's schema.
2. A REFERENCES constraint (foreign key) enforces that:
Correct. A foreign key means "this value must point to a real row in another table." PostgreSQL enforces this on every insert and update — you cannot have a membership for a user that doesn't exist.
Not quite. REFERENCES is a foreign key constraint — it enforces that a value in one table (like user_id) corresponds to a real row in another table (users). UNIQUE handles uniqueness; NOT NULL handles non-empty columns.
3. PostgreSQL's schema enforcement guarantees that:
Correct. Schema enforcement means every row must match the table's defined shape — the right columns, the right types, all constraints satisfied. This is what MongoDB's flexible schema deliberately does not enforce.
Not quite. Schema enforcement means every row inserted or updated must match the table's definition — correct column types, no nulls where NOT NULL is set, unique values where UNIQUE is set. It's a structural guarantee about your data's shape.
