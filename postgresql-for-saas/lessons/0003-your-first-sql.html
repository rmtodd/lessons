Your First SQL

The four essential statements — and the JOIN that changes everything

⏱ ~20 minutes 📖 Read + quiz 🎯 Core language

SQL is the language you use to talk to PostgreSQL. It has been the standard for relational databases since the 1970s and is understood by every analytics tool, BI platform, and data engineer on earth. Learning it is an investment that doesn't depreciate.

SQL is declarative — you describe what data you want, not how to retrieve it. PostgreSQL figures out the execution plan. This is different from writing a MongoDB aggregation pipeline, where you often specify the steps explicitly.

Examples use this SaaS schema

All examples assume a simple SaaS schema: users, organizations, org_memberships, and subscriptions. You'll design this schema fully in Lesson 4.

SELECT — reading data

SELECT retrieves rows from a table. The basic form:

SELECT * FROM users;

* means "all columns." You can name specific columns instead:

SELECT id, name, email FROM users;

Filtering with WHERE

WHERE filters which rows are returned:

SELECT id, name FROM users
WHERE email = 'alice@acme.com';
SELECT * FROM subscriptions
WHERE status = 'active'
  AND expires_at > now();

Ordering and limiting

SELECT name, created_at FROM users
ORDER BY created_at DESC
LIMIT 10;

This reads: "Give me the 10 most recently created users, newest first." DESC = descending (newest first); ASC = ascending (oldest first, the default).

INSERT — adding data

INSERT INTO users (name, email)
VALUES ('Alice', 'alice@acme.com');

You list the columns you're providing; anything with a DEFAULT (like id or created_at) is filled automatically.

To get the new row back immediately:

INSERT INTO users (name, email)
VALUES ('Alice', 'alice@acme.com')
RETURNING id, created_at;

RETURNING is a PostgreSQL extension to the SQL standard — very handy for getting auto-generated IDs back without a second query.

UPDATE — modifying data

UPDATE users
SET name = 'Alicia'
WHERE id = 'a1b2c3d4-...';
Always include WHERE on UPDATE

UPDATE users SET name = 'Alicia' without a WHERE clause updates every single row. Always double-check your WHERE condition before running an UPDATE in production.

DELETE — removing data

DELETE FROM users
WHERE id = 'a1b2c3d4-...';

Same warning as UPDATE — omitting WHERE deletes all rows. In practice, SaaS apps often use soft deletes: add a deleted_at TIMESTAMPTZ column and set it instead of physically removing the row, so you can audit and recover.

JOIN — the relational superpower

This is the capability that most distinguishes PostgreSQL from MongoDB. A JOIN combines rows from two (or more) tables based on a matching condition — in a single query, in a single round-trip to the database.

Example: Find all users who belong to the "Acme" organization, along with their role:

SELECT
  users.name,
  users.email,
  org_memberships.role
FROM users
JOIN org_memberships ON users.id = org_memberships.user_id
JOIN organizations   ON org_memberships.org_id = organizations.id
WHERE organizations.name = 'Acme';

This crosses three tables — users, org_memberships, and organizations — and returns exactly the data you need, shaped for display. In MongoDB, this requires multiple queries stitched together in application code.

How JOIN works

JOIN typeReturnsWhen to use
JOIN (or INNER JOIN)Rows that match in both tablesMost common — use when both sides must exist
LEFT JOINAll rows from the left table, matched or notWhen right side is optional (e.g. users with or without a subscription)
RIGHT JOINAll rows from the right tableRare — usually rewrite as LEFT JOIN from the other direction
FULL OUTER JOINAll rows from both tablesWhen you need unmatched rows from both sides

Example: All users and their subscription status — including users with no subscription yet (LEFT JOIN):

SELECT
  users.name,
  users.email,
  subscriptions.status  -- NULL if no subscription
FROM users
LEFT JOIN org_memberships ON users.id       = org_memberships.user_id
LEFT JOIN subscriptions   ON org_memberships.org_id = subscriptions.org_id;

Retrieval practice — 3 questions

Close your eyes for a moment, then answer from memory. The harder it feels, the more it's sticking.

1. Which SQL keyword is used to retrieve data from a table?
Correct. SELECT is the read operation — equivalent to MongoDB's find(). The other three (INSERT, UPDATE, DELETE) modify data.
Not quite. SELECT is the read statement in SQL. INSERT adds rows, UPDATE modifies them, DELETE removes them. Think of CRUD: Create → INSERT, Read → SELECT, Update → UPDATE, Delete → DELETE.
2. What does the WHERE clause do in a SELECT statement?
Correct. WHERE filters rows — like MongoDB's query selector (the object you pass to find()). Columns are chosen in the SELECT list; ordering uses ORDER BY; limiting uses LIMIT.
Not quite. WHERE filters rows — it's the condition that determines which rows come back. To choose columns, list them after SELECT. To sort, use ORDER BY. To limit row count, use LIMIT.
3. A JOIN allows you to:
Correct. JOIN merges rows from two tables based on a matching condition (like user_id = users.id). This is the key operation that document databases require application code to replicate with multiple queries.
Not quite. JOIN combines rows from multiple tables in a single query. This is the central advantage of relational databases — relationships between entities are queryable directly in SQL, without multiple round trips to the database.
