Phase 2 — Live-Database Recon

✅ DONE — completed 2026-07-03, findings written up in docs/background/phase2-live-db-recon.md.

The one-sentence version

Before writing any code that changes the real database, go look at the actual, live database directly and verify every assumption the plan was built on — because a plan written from memory, old docs, or a quick skim of the code will always contain a few wrong guesses, and it's far cheaper to find those now than after a migration has already shipped.

Why this phase exists

Every other phase in this plan assumes specific facts about the database: which columns already exist, which don't, what type an ID column is, whether a security feature is already on or off. Phase 2's entire job was to stop trusting the plan's early drafts and go check, directly against the live database, item by item.

Analogy This is a surveyor visiting the actual construction site before the architect finalizes the blueprint — not to redesign the building, but to confirm the ground really is level where the drawings assumed it was, and to flag the one spot where there's already a gas line nobody's blueprint accounted for.

This matters more than it might sound like, because several later phases in this plan explicitly say things like "recon-corrected targets... the plan's old line-refs were stale" (Phase 4) — meaning: without this phase, later work would have quietly written code against wrong assumptions, and either failed outright or silently done the wrong thing.

What was actually checked, and what it changed

Eight separate corrections came out of this recon and were folded into Phases 3 through 5. The most consequential ones:

ID column types (primary keys) across tablesNot fully knownsec_filings uses a plain integer ID — every other table uses uuid or bigintThe override system's ledger (Phase 4) has to store IDs from many different tables in one shared column, then cast them back to the right type when writing — get this wrong and a single override RPC silently fails on that one table.

The habit worth noticing This phase is a specific instance of a much more general rule: verify against the live system before building on top of an assumption about it, especially when a mistake would touch real production data. It shows up again explicitly in Phase 4 and Phase 5 ("verify the RLS migration on a seeded throwaway branch before live").

How the recon was actually done, without touching production

The recon read Postgres's own internal "catalog" tables (pg_policies, information_schema) — think of these as the database's own self-description, a table that lists all the other tables, their columns, and their security rules. Reading these is completely safe; it's just looking, not changing anything.

Check your understanding

Why couldn't Phase 2 just trust the plan's first draft and the existing code comments?
Nothing here is about anyone lying — it's just that documentation, comments, and plans are static snapshots, while a live production system keeps changing underneath them. Phase 2's job was closing that gap before betting expensive migration work on stale assumptions.

Recommended primary source

Supabase Docs — Row Level Security. This is the concept the recon spent the most effort verifying (RLS state across every table), and understanding it now sets up Phase 5 cleanly.

If any of the table names or "recon-corrected" details feel unclear, ask your teaching agent — this lesson deliberately kept the recon findings compressed to the ones that shaped later phases.
