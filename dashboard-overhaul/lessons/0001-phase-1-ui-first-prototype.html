Phase 1 — The UI-First Prototype

✅ DONE — completed 2026-07-03, commit 4420e09 on branch feat/dashboard-app-prototype.

The one-sentence version

Before writing a single line of "make this button actually do something" code, build the entire new dashboard's screens, using real data, with every button that would change something showing a "not wired yet" message instead of actually changing it — so you (Raymond) could click through the whole thing and approve the layout before anyone spent a week wiring up real logic behind buttons that might get rearranged anyway.

Why this phase exists — the problem it prevents

The most expensive mistake in rebuilding an internal tool like this dashboard isn't a bug — it's building the wrong thing fully, then having to tear it apart because the layout doesn't match how you actually work. The team that owns this plan explicitly chose to front-load that risk: build the skeleton first, get it rejected or approved cheaply, and only pay the much higher cost of wiring real backend logic once the shape is locked in.

This is why the plan's decision log (§2 of the plan) says: "User-confirmed sequencing: UI first... all writes are visibly stubbed." A stub is a button that looks completely real and clickable but, instead of changing anything, just pops up a small toast message like "not wired yet." That's deliberate — it lets you evaluate the shape of the tool (is this the right screen, are these the right columns, is this the right home page) completely separately from whether the underlying logic is correct yet.

Analogy This is like a contractor building a full-size cardboard mockup of a kitchen renovation — every cabinet, counter, and appliance in its exact place — before pouring a single pipe or wire. You walk through the cardboard kitchen and say "actually, move the fridge over there" for free. Nobody wants to discover the fridge is in the wrong place after the plumbing's already run behind the wall.

What actually got built

All 8 top-level screens ("routes," in web-development terms — see the SPA glossary entry) of the new dashboard were built as real, clickable pages:

Triage (home)  ·  Deals  ·  Documents  ·  15G Review
Sources (Presales / Freddie / EDGAR)  ·  Costs  ·  Logs  ·  Settings

Two important choices baked into this phase, both from the plan's §2 "Decisions" section:

  • Reads are real, writes are fake. Every screen actually pulls live data straight from the Supabase database (using the same "read-only public key" approach the old dashboard already uses today — a proven, working pattern, not a new risk). But clicking "rename this deal" or "confirm this match" just shows the stub toast. This split is possible because reading data safely is a solved, low-risk problem here; writing data safely (with the override system, actor attribution, etc.) is the hard, high-risk part that later phases handle properly.
  • Not server-rendered. The plan explicitly rules out a framework called "TanStack Start" that would need its own always-running Node.js server process. Instead this is a plain SPA — static files served by a simple web server (nginx), with the existing Python backend (FastAPI) doing all the real work. One fewer moving service to run and monitor in production.

What Raymond's review actually changed

This phase wasn't a one-shot build — it went through two rounds of walkthrough feedback from Raymond, and several real decisions got locked in because of that review (captured in the plan's progress log, §0, and decisions §2):

  • The old "Pipeline" tab was replaced entirely by a new Triage home page — a "here's what needs your attention right now" work queue, instead of a passive status view.
  • A new concept, the human review tracker, was introduced at Raymond's request (originally called "validation," renamed after discussion): every machine-made match or classification now carries a state — unreviewed, confirmed, corrected, or dismissed — with comments. This became a load-bearing concept for several later phases (see Phase 4's confirm/dismiss).
  • The visual scheme for "which rating agencies have coverage on this deal" (colored dots: blue = matched, green = downloaded, gray = still looking, hollow = stopped) was fixed during review.
  • A real bug was caught: the "Monitoring" status column could show something that contradicted the "Status" column, because monitoring didn't respect whether a deal was actually active. Fixed before backend work ever started on it.
Why this matters for later lessons Every one of these decisions became a fact baked into the plan for the phases that follow. When you read "Triage" or "review tracker" in a later phase, it's referring back to something decided here, on cardboard, before any real code risk was taken.

Check your understanding

Why did the team make writes fake ("stubbed") in Phase 1 but reads real?
The point isn't "reads are easy, writes are hard" as a universal rule — it's that this specific team already had a proven, working way to read data safely, and no proven way yet to write it safely (with human overrides protected). Faking the writes let them test the UI's shape immediately, without gating that on the hardest, highest-risk part of the whole plan.
