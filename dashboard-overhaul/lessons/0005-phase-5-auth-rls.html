Phase 5 — Auth + RLS Groundwork

← NEXT — not started yet as of 2026-07-03. Estimated ~1–1.5 days (smaller than originally scoped, because Phase 4 already built the actor-attribution plumbing this phase needs).

The one-sentence version

Turn on real login for the new dashboard, and turn on the database's own security rules (so it enforces "only logged-in people can read this" itself, rather than trusting every connected program to behave) — without breaking the still-running old dashboard while this transition happens.

Why this phase exists

Recall from Phase 2's recon: Row Level Security (RLS) is off on almost every table today. That means anyone holding the public "anon" read-only key — which the old dashboard has always used, openly, in the browser — can read essentially the entire database directly. That's been an acceptable risk so far because the only thing using that key is the trusted old dashboard. But the new dashboard is about to expose sensitive new tables (the override ledger, with human comments and audit-log entries, actor names) that were never meant to be casually readable by anyone with that key.

Analogy Imagine an office where the front door has always been unlocked, but the only person who's ever walked in is a trusted coworker. That's been fine. Now you're about to add rooms with sensitive personnel files inside — you can't keep relying on "well, nobody's tried the door" as your security model once there's something worth stealing behind it. You lock the door and hand out keys only to people who log in.

Two separate things happening at once (don't conflate them)

This phase covers two related but distinct concepts — worth keeping straight, because they answer two different questions:

ConceptQuestion it answersEnforced where
Auth (JWT-based login)"Who is this person?"The FastAPI backend, checking a signed token attached to every request
RLS (Row Level Security)"What is this person allowed to read, at the database level?"Postgres itself, on every single query, regardless of which program is asking

You genuinely need both. Auth alone would only protect requests that go through the FastAPI backend — but the new dashboard's read path (per the plan's §2 decisions) goes directly from the browser to Supabase for most reads, bypassing FastAPI entirely for speed. If only the backend checked identity, all those direct reads would remain wide open no matter how good the login screen looks. RLS is what makes the database itself refuse to hand out rows to someone who isn't allowed to see them, independent of which door they came through.

The "swap the stub" trick — most of the hard part is already done

Here's a pleasant surprise buried in this phase: Phase 4 already built all the actor-attribution plumbing this phase needs — a placeholder function called require_actor already sits on every write-changing endpoint, and the RPCs already carry an actor's identity all the way into the audit log. Today, that placeholder just trusts a plain header sent by whatever's calling it (or defaults to a generic "dashboard" actor) — it doesn't yet verify a real, cryptographically signed login token.

Phase 5's actual job on the backend side is much smaller than it sounds: swap the inside of that one placeholder function to actually verify a real Supabase login token (a JWT), instead of trusting whatever header shows up. Everywhere that placeholder is already wired in (every mutating endpoint, from Phase 4) automatically gets real protection the moment the swap happens — no new wiring required at each of those call sites.

Why front-loading Phase 4's plumbing paid off This is a direct payoff of Phase 4 having been built with auth in mind from the start, even though auth itself wasn't wired yet. It's a good example of a general engineering habit: build the "who is doing this" plumbing early and generically, even before you've decided exactly how identity will be verified — swapping the verification logic later is far cheaper than retrofitting attribution into dozens of already-shipped endpoints.

There's also a safety valve: a setting called REQUIRE_AUTH, defaulting to false. Until it's flipped on (planned for Phase 9, cutover), the backend keeps accepting the old trust-a-header behavior — so this phase can ship without immediately locking anyone out while the frontend login screen is still being built and tested.

The risky part: turning RLS on without an outage

This is the part the plan is most careful about, for good reason: turning RLS on with the wrong policy is an instant outage. If a table's new security policy is even slightly too strict, the still-running old dashboard (which reads with the plain anon key, not a real login) instantly loses access to real data, in production, with real people relying on it.

The plan's mitigation has two parts:

  • Two policies exist side-by-side, temporarily. Each table gets both an authenticated policy (for the new, logged-in dashboard) and a temporary anon policy that deliberately replicates exactly what the old dashboard could already see — so nothing changes for the old dashboard's access pattern, even though RLS is technically now "on." The anon policies get removed later, at Phase 9's cutover, once the old dashboard is retired for good.
  • Test on a disposable copy of the database first. Following the same instinct as Phase 2's recon and Phase 4's verification approach, this migration gets tried on a seeded, throwaway Supabase branch (a temporary, isolated copy of the database) before ever touching the live one.

A newly-discovered wrinkle that shapes this phase's scope

Phase 4 surfaced a real, live security gap worth understanding, because Phase 5 has to correct for it directly: two brand-new internal tables built in Phase 4 — the override ledger and the audit log — were, by default, readable by anyone with the anon key, the same as everything else. That's a problem specifically because these tables now contain human review comments and actor names that were never meant to be casually public. The plan is explicit: give these tables authenticated-only policies from the very start — do not replicate the old dashboard's anon access to them, since the old dashboard never needed to see them anyway.

A second, subtler gotcha the plan flags: simply running REVOKE ... FROM PUBLIC on a database permission does not remove Supabase's own default grants to the anon and authenticated roles — those are separate, additional grants Supabase adds automatically. This actually bit Phase 4 for real: the override-writing RPCs were briefly callable by anyone with the anon key, directly over Supabase's REST API, despite looking properly locked down. The fix had to explicitly name anon, authenticated in the revoke statement. Phase 5 has to re-run Supabase's own automated security-advisor check after the RLS migration specifically to catch this class of mistake again.

Check your understanding

Why isn't backend-side login (Auth/JWT) enough by itself, without also turning on RLS at the database?
The key fact is the plan's read-path decision from §2: reads go straight to Supabase, not through FastAPI. Any protection that only lives in FastAPI would have a wide-open hole exactly where most traffic flows.
