Phase 9 — Parity + Cutover

Not started — planned duration ~2–3 days, followed by 48 hours of observation. This is the last phase — after it, the new dashboard is the dashboard, and the old one is gone.

The one-sentence version

Confirm the new dashboard can do everything the old one could, flip on real security for good, permanently retire the old dashboard, and clean up every piece of the migration's temporary scaffolding — then watch closely for two days before calling it done.

Why a dedicated "cutover" phase, instead of just... switching?

Every earlier phase in this plan was built to run alongside the old dashboard without disturbing it — reads use the same proven anon-key pattern; RLS was rolled out with temporary anon policies specifically so the old dashboard wouldn't notice anything changed; auth had a REQUIRE_AUTH=false escape hatch. That caution was deliberate, and it means the moment of actually retiring the old system is its own distinct, risk-bearing event — not something that happens automatically as a side effect of earlier phases. Phase 9 exists to make that moment small, checked, and reversible-feeling, rather than a single irreversible leap.

Analogy Moving into a newly renovated house while still keeping the old apartment's lease active, utilities running, and a spare key — right up until the day you've actually walked every room of the new place, confirmed the plumbing works, and only then hand back the old apartment's keys for good. Phase 9 is the "hand back the keys" day, not the renovation itself.

What "parity" actually means, concretely

Before switching anyone over, the plan calls for an explicit parity checklist — walking through everything the old, 7,029-line dashboard could do, and confirming the new one covers it (or that its absence was a deliberate, already-approved decision, like dropping the dead T1/T2/T3 columns back in Phase 1). This matters because a rebuild that's 95% feature-complete but silently missing one thing someone relies on daily doesn't feel like progress to that person — it feels like a regression, even though the other 95% is strictly better.

The security flip, now for real

Recall the safety valve from Phase 5: REQUIRE_AUTH defaulted to false, so the backend kept accepting its old, looser trust-a-header behavior while the rest of the system caught up. Phase 9 is where that finally flips to true — from this point on, every mutating request genuinely must carry a real, verified login token; there's no more fallback. This is only safe to do now because parity has been confirmed and the old dashboard (the one thing that never had real login) is being retired in the very same phase.

Cleaning up the migration's own scaffolding

A less obvious but important part of this phase: several things were deliberately kept around during the transition purely to avoid breaking the old dashboard, and now that it's gone, they can finally be removed:

What gets removedWhy it existed at all, and why it's safe to remove now
The temporary anon RLS policies (Phase 5)They existed only to keep the old dashboard's anon-key reads working during the transition. With the old dashboard gone, nothing uses the anon key for these tables anymore.
The old 15G "*_override" columns (deal_name_override, deal_type_override, etc.)These were the original, pre-Phase-4 way humans corrected 15G data — kept alive only because the old dashboard still read and wrote them directly. Phase 4 already copied their real historical values into the new ledger and wrote the effective values through to the base columns, so by this point they're pure dead weight, safe to drop.
presale_deal_override_log (the old, single-purpose presale override table)Its data was backfilled into the unified ledger back in Phase 4 (marked source='backfill'). Frozen (stopped being written to) once Phase 4 landed; now dropped entirely.
app_preferences tableIts two remaining keys migrate into the new app_settings table from Phase 6.5, then the old table is dropped — one settings system, not two.
The pattern worth remembering Notice how many of these cleanups were already planned back when the thing being replaced was first built (Phase 4 explicitly says "keep columns until old dashboard retires, then drop"). Good migrations don't just build the new thing — they leave themselves a clear, already-agreed-upon todo list for removing the old thing's remnants, instead of leaving "temporary" scaffolding in place indefinitely, which is how codebases quietly accumulate permanent-feeling temporary cruft.

Why 48 hours of watching, after everything already passed its checks

Even after parity is confirmed and every automated check passes, the plan calls for 48 hours of just observing the running system (workers, Slack integration) before considering the cutover truly finished. This acknowledges something automated tests can't fully cover: some problems only show up under real, live, slightly-unpredictable production traffic over time — a batch job that only runs once a day, a rare filing type, an edge case in a scraper's retry logic that only surfaces after enough real-world attempts. This is a healthy, humble habit: "the tests passed" and "this is safe in production" are related but not identical claims, and the gap between them is exactly what a short observation window is for.

Looking back at the whole plan from here

Standing at Phase 9, it's worth noticing the shape of the whole journey these ten lessons covered:

  • Phases 1–2 were about not building on wrong assumptions — first about the UI (validated with real review, cheaply, before real backend risk), then about the database itself (recon before any migration).
  • Phases 3–4 built the two foundational systems everything else depends on: honest cost accounting, and the override/actor system that protects every human correction from every later phase forward.
  • Phase 5 added real security, carefully, without an outage.
  • Phases 6–8 connected the already-safe pieces together into a fully working product, closing small gaps as they became visible.
  • Phase 9 confirmed nothing was missed, flipped on full security, and — importantly — cleaned up every piece of temporary scaffolding the transition itself required.

That's a genuinely disciplined shape for a large rebuild: validate cheaply and early, build the hard safety-critical foundations before the features that depend on them, and treat "removing the old system's remnants" as real, planned work — not an afterthought.

Recommended primary source

No single external doc covers "cutover phase design" as one topic — this is standard, well-established software-migration practice assembled from experience. If useful outside grounding, Martin Fowler's writing on the "Strangler Fig" migration pattern describes the broader strategy this whole plan follows: build the new system alongside the old one, route more and more real work to it gradually, and only remove the old system once nothing depends on it anymore.
