Phase 6 — Wire Writes

Not started — planned duration ~1 week, comes after Phase 5.

The one-sentence version

Go back to every "not wired yet" stub button built in Phase 1's clickable prototype, and replace each toast message with a real call to the real endpoint — most of which were already built back in Phase 4, sitting unused until now.

Why this phase exists, and why it's straightforward now

This is the payoff of the whole "UI-first" strategy from Phase 1. Back then, the team deliberately built every screen with fake writes so the layout could be reviewed cheaply. Now that the layout is locked in (Phase 1), the database structure and protection rules exist (Phase 4), and login/security are in place (Phase 5), this phase is comparatively low-risk: connect button to endpoint, surface-by-surface, removing one stub toast at a time.

Analogy Back to the cardboard-kitchen analogy from Phase 1: this is the phase where the real plumbing finally gets run behind the walls that were already approved. Because nobody's still arguing about where the fridge goes, the plumber can just plumb.

The plan is explicit that most of the actual REST endpoints already exist from Phase 4 — /overrides, /overrides/{id}/revert, /deals/{id}/rename, /edgar/15g/{id}/reclassify, /presale-index/{id}/review, and more. This phase is largely about the frontend side: swapping each stub's fake toast for a real network call, and handling what happens when that call succeeds or fails on screen (see optimistic update in the glossary).

What's genuinely new here (not just wiring)

A few things in this phase are actual new work, not just connecting existing plumbing:

  • GET /triage, GET /me, and GET /meta/field-registry go live — the endpoints Triage and the override UI need to function for real, now enriched with the reviewed/total counts the Phase-1 "Review queue" strip needs.
  • Force re-check gets its actual migration and endpoint (see Phase 12 in the plan, §12, for the detailed mechanics) — this is the feature that lets a human bypass the scraper's normal stopping rules (like "we already found 3 agencies' worth of coverage, stop looking") for one deal, one time.
  • A new small feature: deal notes. A plain freeform annotation a human can attach to a deal — new table, two simple endpoints. Small, but a good example of "not everything needs the override system" — notes aren't correcting a machine value, so they don't need a ledger entry, just a normal table.
  • /presale-index/{id}/retry-download — resets the download-attempt counter for presales that either failed or simply expired, so they can be retried instead of sitting stuck forever.

Two follow-up corrections that surfaced only once the pieces were connected

This is a useful, honest pattern to notice: some problems only become visible once you actually wire two previously-separate pieces together, even though each piece worked fine in isolation.

  • Excluded/dismissed rows were leaking into lists. Phase 4 added excluded and dismissed flags so a human could mark a deal or filing as irrelevant. But the actual list screens (Deals, 15G Review, Presales) and the /triage endpoint weren't yet filtering those rows out — so something a human explicitly marked "not relevant" would still show up everywhere. This phase fixes that filtering, surface by surface.
  • The reclassify/exclude cascade isn't atomic yet. Recall from Phase 4's glossary entry on cascades: reclassifying a deal's type has knock-on effects (updating monitoring flags, reconciling monitoring tasks). Today those happen as several separate database calls in a row, not one atomic unit — so a crash mid-sequence could leave a deal in an inconsistent state (reclassified, but monitoring flags not yet updated to match). This phase's plan calls for a proper all-or-nothing RPC that does the whole cascade as one transaction, closing that gap.
Why this is worth noticing, not just fixing Neither of these was a mistake in Phase 4 — they were reasonable to defer, because they only become observable problems once the real UI and real cascade logic are exercised together. This is a normal, healthy part of phased work: each phase surfaces a small number of "oh, now that we can see it end-to-end, here's a gap" items for the next phase to absorb.

The registry's second job, reinforced here

One rule from Phase 4 gets repeated explicitly in this phase's notes, because it matters every time a new field gets added from here on: any newly overridable field needs one line added to the field registry. The registry isn't just UI decoration — the generic /overrides endpoint actively rejects any attempt to write a field that isn't registered. So this phase's "wire everything up" work has a hard dependency on the registry already existing and being complete.
