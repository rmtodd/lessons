Phase 6.5 — Admin Settings

Not started — planned duration ~3–4 days. Notice the "6.5" numbering: this phase was inserted into the plan after the original numbering was set, because the team recognized a real gap mid-plan (see below) — the half-number just means "this slots in right after Phase 6, added later."

The one-sentence version

Move the dozen-plus behavioral knobs that currently live as Railway environment variables (frozen in place until someone redeploys the whole application) into the database instead, so an admin can tune them — like "how often should we check for new presales" or "which AI model should extract 15G filings" — directly from the Settings screen, and have the change take effect within about 30 seconds, with no deploy at all.

Why this phase exists

Today, every behavioral setting in CMBSDOCS — how often the presale scraper runs, whether it should run on weekends, which specific AI model does 15G extraction — is a Railway "environment variable": a piece of configuration baked in at the moment the application starts up. Changing any of them means editing that configuration and redeploying the entire application — a real, if small, amount of downtime risk and delay, for what's often a one-line behavioral tweak.

Analogy Today, changing the thermostat setting means turning off the whole building's power, rewiring the thermostat, and turning the power back on. This phase installs an actual thermostat dial on the wall — the building stays running the whole time, and the new setting takes effect within moments.

Crucially, the plan draws a firm line about what doesn't move: actual secrets (API keys, encryption keys, database URLs) and safety-critical settings (like whether authentication is required at all) stay in Railway, never in the database. This phase is only about behavioral knobs — schedules, feature toggles, model choices — not security posture or credentials. That's a deliberate, important boundary: a setting whose whole point is "must survive even if the database is down or compromised" (like REQUIRE_AUTH) cannot itself live inside the thing it's meant to protect against.

How the "no redeploy needed" trick actually works

The recon behind this phase (done as part of this same phase's planning) found something useful: every background loop in the system already re-reads its own "am I enabled" flag and its own interval fresh on every single cycle — nothing in the existing code assumes those values are fixed forever. The only place a setting actually gets "frozen" is a handful of one-time startup checks like if enabled: create_task(...) — meaning "only ever start this background loop at all if the setting was true the moment the app booted."

So the actual engineering change is smaller than it sounds:

  • Delete those one-time startup gates — always start every loop, unconditionally.
  • Add a small helper, get_setting(key, default, cast), that checks (in order): is there a row for this key in the new app_settings table? If not, fall back to the existing environment variable. If that's also absent, fall back to the code's built-in default.
  • When a loop is "disabled" via this new setting, instead of not running at all, it just sleeps for a short, capped interval (30 seconds or its normal interval, whichever is shorter) and checks again — so flipping it back on is felt almost immediately, not after the next full interval.
Why a 30-second cache, not instant Rather than have every single loop iteration hit the database to check a setting, the helper caches each value in memory for 30 seconds per running process. This means a change can take up to 30 seconds to be felt by any one process, and — since there are multiple separate processes running (the web server, the scheduler, the workers) — they might briefly disagree with each other for a few seconds. The plan accepts this small, bounded staleness deliberately, because it avoids hammering the database with a query on every loop tick, and 30 seconds of lag is a non-issue for settings like "should we scrape on weekends."

There's also an explicit fail-open rule: if the database is briefly unreachable when a setting is checked, the code logs a warning and falls back to the already-known-good value (env var or default) rather than crashing the loop. A settings outage should never take down the whole ingestion pipeline.

Why this needs its own registry too (déjà vu, on purpose)

Just like Phase 4's field registry controls which data fields can be overridden, this phase adds a parallel SettingSpec registry — one entry per setting, describing its type, valid range, and default. The Settings screen renders its whole form generically off this registry, the same way the override system's UI renders generically off the field registry. Recognizing "we've already solved this shape of problem once" and reusing the same pattern, rather than inventing a new one, is a mark of a well-organized codebase.

Check your understanding

Why does the plan insist that things like `REQUIRE_AUTH` and database credentials stay in Railway environment variables, never in this new database-backed settings system?
This is a "who guards the guards" problem: a security-critical setting has to live somewhere independent of the exact systems it's meant to protect against failing. Behavioral tuning knobs (schedules, model choices) don't have that constraint, which is exactly why they're the ones that moved.

Recommended primary source

There isn't a single canonical external doc for "runtime feature flags with env-var fallback" — it's a common but informally-documented pattern. The closest useful primary-source framing is Martin Fowler's classic article "Feature Toggles (aka Feature Flags)", which covers the general idea of runtime-configurable behavior this phase is a specific instance of.

If the "DB > env > default" precedence order feels fuzzy, ask your teaching agent to trace one specific setting (e.g. the 15G extraction model) through all three fallback layers with concrete example values.
