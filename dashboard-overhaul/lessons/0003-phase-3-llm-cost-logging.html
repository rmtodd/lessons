Lesson 3 of 10 · Dashboard Overhaul

Phase 3 — Central LLM Cost Logging

✅ DONE — completed 2026-07-03, on branch updated-dashboard-stage2.

The one-sentence version

Make every single call to an LLM (the AI models that read filings and extract deal data) automatically record what it cost, in one place — so the future "Costs" screen in the dashboard can show, truthfully and completely, where the money is going, instead of only seeing whatever a few scattered manual logging calls happened to remember to record.

Why this phase exists

CMBSDOCS uses paid LLM calls throughout its pipeline — extracting structured data from 15G filings, matching presale documents to deals, and more. Each call costs real money, based on how many "tokens" (roughly, chunks of text) go in and come out. Before this phase, that spend was tracked manually: a handful of specific places in the code happened to write a row into a table called api_call_log after a call finished — but plenty of call sites didn't, and there was no shared, consistent way to compute the actual dollar cost from the raw token counts, especially since different AI providers (OpenAI, Gemini) charge differently and structure their "reasoning" tokens differently.

Analogy Before this phase, expense tracking worked like a company where employees are supposed to fill out a paper receipt for each business lunch — reliable? Only when everyone remembers, and only if the receipt actually shows the right total. This phase installs a company card that automatically logs every purchase, at the point of purchase, with the true cost calculated the same way every time.

How it actually works

The team built one wrapper, LLMCallContext, that describes what a given LLM call is for (which operation, which deal, which filing) before the call happens. Then a single shared function, call_json(..., context=...), wraps the actual call to the AI provider and — regardless of whether the call succeeds or fails — writes exactly one row into api_call_log afterward, computing the true cost via a new provider-aware pricing function (pricing.estimate_cost_usd) that correctly handles each provider's different token-pricing rules (e.g. OpenAI already includes "reasoning" tokens in its completion count; Gemini needed that added on separately).

This was wired into all 6 real call sites across the codebase in one pass, and the 3 old, manual, inconsistent logging helpers were deleted — meaning there's now exactly one way this happens, not four slightly different ones that could drift apart.

A design choice worth noticing: "best-effort" logging The plan is explicit that this logging must never fail an extraction — if writing the cost-log row itself has a problem, the actual extraction work still has to succeed. Logging is a side-effect, not a dependency. This is a recurring, important pattern: instrumentation/observability code should never be allowed to break the feature it's observing.

The "estimated" spend problem

Not every cost is a clean, per-call LLM API charge. Two other tools cost money without giving a clean per-call receipt: Browser-Use (an AI agent that operates a real web browser to log into agency sites) and Firecrawl (a web-scraping service). For these, the plan adds explicit estimate rows — clearly labeled estimated=true in their metadata — computed from a configured "average cost per task" rather than an exact receipt. The Costs screen (Phase 8 fleshes this out) is expected to show these separately from the precise, receipt-backed LLM costs, so nobody mistakes a rough estimate for an exact number.

Why this had to happen before the Costs screen could be trusted

The plan's §4.7 describes a "Costs" tab with real charts and anomaly alerts (e.g. "today's spend is 3× the trailing 7-day median"). None of that is meaningful if the underlying log is incomplete or inconsistently computed — an anomaly detector built on top of half-recorded data would either miss real problems or cry wolf constantly. Phase 3 is the unglamorous prerequisite that makes Phase 8's cost dashboard trustworthy at all.

What was deliberately left for later

A full live end-to-end test (actually running a real 15G re-extraction and watching a new row land in the real production table) was not done in this phase — the note in the plan says the local development environment this was built in doesn't have the real Supabase/Redis infrastructure installed. Instead, the team verified the database structure changes ("schema round-trip") directly. This is a normal, honest trade-off: verify what you can verify safely, and be explicit about what's still unverified rather than pretending it's fully tested.

Check your understanding

Why does the plan insist that cost-logging must never cause an extraction to fail, even if the logging itself breaks?
This is a general principle, not specific to this codebase: anything whose entire job is to *observe* a system (logs, metrics, cost tracking) should be built so that its own failure can never take down the thing it's observing.

Recommended primary source

There isn't one canonical external doc for "LLM cost tracking design" — this is a project-specific pattern. If you want outside context on how token-based LLM pricing works generally, OpenAI's own Text generation guide explains what "tokens" and "reasoning tokens" mean, which is the raw unit this whole phase is converting into dollars.
