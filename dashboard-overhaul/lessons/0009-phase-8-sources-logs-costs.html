Lesson 9 of 10 · Dashboard Overhaul

Phase 8 — Sources Actions + Logs/Costs Endpoints

Not started — planned duration ~1 week.

The one-sentence version

Finish wiring up the remaining "Sources" screens (Presales, Freddie, EDGAR) with their real action buttons, extend the audit-log trigger to a few tables that were missed, add per-run scrape history and calendar snapshots so agency health is actually diagnosable, and finish the Costs dashboard's remaining backend pieces.

Why this phase exists

This phase is best understood as "closing several small, specific, previously-identified gaps," rather than one single new idea — it's a cleanup-and-completion phase. Three separate gaps, each worth understanding on its own:

Gap 1 — some tables still have no audit trail

Recall from Phase 2's recon: the presale index and Freddie index tables had no audit log trigger at all. Phase 4 built the actor-attribution machinery (the RPCs that carry "who did this" into the database), but the actual trigger that writes audit-log rows was never extended to fire on these two tables (plus app_preferences and the new app_settings table from Phase 6.5). The plan is precise about why this matters concretely: today, a human unlinking a presale match or dismissing one is captured in the override ledger (so the "what changed" story exists), but not in the audit log (so the older, simpler "raw before/after" trail is incomplete for those actions) — until this phase's trigger extension closes that gap. Once it does, the actor-attribution plumbing that's already flowing through the RPCs starts working on these tables automatically, with no further wiring needed.

Gap 2 — "why is this agency's scraper stuck?" has no good answer today

When a presale-scraping run has partial failures — some documents matched, some downloads failed with specific errors — that detail mostly vanishes once the run finishes; the "Agency health" cards in the old Sources screen approximate it client-side from whatever's currently in the database, rather than from an actual historical log of what happened on each run. This phase adds a real scrape_runs table capturing, per agency, per run: when it ran, how many rows matched, how many failed, and the specific errors — served through a new endpoint the health cards' "view log" dialog can call directly, replacing the old approximation with ground truth.

Gap 3 — Freddie's calendar page changes silently, with no record of what it used to say

Freddie Mac's issuance calendar is fetched from a PDF that changes over time as deals move through their process. Today, nothing records what that PDF looked like at any earlier point — so if an entry's listed date or status changes, there's no way to see what it said before. This phase adds a snapshot: when an entry's details actually change, the calendar PDF at that moment gets captured and stored, retrievable per entry later.

What ties these three together

Notice the shared shape: in each case, something was working (the scraper runs, the calendar gets polled, the presale index updates), but there was no durable record of what happened along the way when things went wrong or changed — only the current end state. This phase is fundamentally about observability: giving a human (or a future debugging session) enough historical detail to answer "what actually happened here, and when" without having to guess from the current snapshot alone.

Analogy A car's dashboard telling you the engine is "fine" right now isn't as useful as a black box recorder that can tell you exactly what happened three minutes before the warning light came on. Phase 8 is mostly installing black boxes on parts of the system that previously only reported their current state.

Also finished here: the last pieces of the Costs screen

Phase 3 built the actual cost logging — every LLM call recording its true cost consistently. Phase 8 finishes the reporting side: a rollup database view (v_llm_cost_daily, pre-aggregating cost by day/operation/model/provider so the Costs screen doesn't have to scan every raw row every time it loads), plus the export and failed-jobs endpoints the Logs screen needs. This is a good moment to notice how far apart in the plan a feature's pieces can land — the Costs screen's foundation was Phase 3, and it isn't fully wired until Phase 8, five phases later.

Check your understanding

What's the common theme across the audit-trigger extension, scrape-run logging, and calendar snapshots in this phase?
None of these three add a brand-new capability the pipeline didn't have before — they add the ability to *look back* at what happened, which is exactly what makes debugging a "why is this stuck" question possible instead of a guessing game.
