Glossary — CMBSDOCS Dashboard Overhaul

Every term used across the phase lessons, defined once here. When a lesson uses one of these words, it links back to this page instead of re-explaining it. Terms are grouped by the concept cluster they belong to, not alphabetically, so related ideas sit together.

Databases & Postgres

Database
A structured place to permanently store information as rows in tables — think a very large, very strict spreadsheet that many programs can read and write to at once. CMBSDOCS uses Supabase, which runs Postgres underneath.
Postgres (PostgreSQL)
The specific database engine CMBSDOCS's data lives in. When the plan says "a migration on Postgres" or "a Postgres trigger," it means: a change to the rules of this database.
Table / Column / Row
A table is one spreadsheet-like grid (e.g. deals, edgar_15g_filings). A column is one named field every row has (e.g. deal_name). A row is one record — one specific deal, one specific filing.
Migration
A saved, versioned file containing the exact instructions to change the database's structure (add a table, add a column, add a rule) — never a live, undo-able edit made by hand. Every migration is a permanent, ordered step in the database's history, like a numbered patch note. This exists so the exact same change can be replayed on a test copy of the database before it's ever run on the real one, and so there's a paper trail of every structural change ever made.
AnalogyRenovating a building by filing a permitted blueprint for every wall you move, in order, rather than just knocking walls down and hoping you remember what you did.
Primary source: Supabase Docs — Database Migrations
JSONB / JSON blob
Instead of every piece of data getting its own column, some columns hold a whole nested bundle of data as one flexible unit (JSONB). It's useful when the shape of the data varies row to row or changes over time (e.g. an LLM's extracted attributes, which grow new fields as the extraction schema evolves) — but it's harder for the database to protect or query than a plain column. The plan calls one nested location inside a JSONB blob a field path (e.g. extracted_attributes.deal.trust_name). Primary source: PostgreSQL Docs — JSON Types
Trigger
A rule attached directly to a database table that automatically runs a small program every time a row is inserted, updated, or deleted — no matter which outside program made the change. Triggers are the database's own immune system: they fire even if the code that changed the data forgot to follow the rules, or wasn't written to know about them. CMBSDOCS uses two: one that snapshots every change into the audit log, and one (the reassert trigger) that puts a human's override value back if a machine process tries to overwrite it. Primary source: PostgreSQL Docs — Triggers
RPC (Remote Procedure Call)
A named, pre-written bundle of database steps that a program can call as a single unit — e.g. app_apply_override(...). It exists because some actions need several things to happen together, atomically (see atomic), and a single plain read/write can't guarantee that. RPCs in this plan also carry the "who did this" information (see actor) into the database for that one call.
Atomic (as in "atomic write")
A set of database changes that either all happen or none happen — there's no in-between state where only half of it landed. This matters when one user action (like "reclassify this deal") needs to update several tables together; without atomicity, a crash mid-update could leave the data in a broken, inconsistent state.
RLS (Row Level Security)
A Postgres feature that lets the database itself decide, row by row, who is allowed to read or write it — instead of trusting every program that connects to behave. Today almost every CMBSDOCS table has RLS off, meaning anyone with the read-only "anon" key can see everything. Phase 5 turns RLS on for the tables the new dashboard reads directly. Primary source: Supabase Docs — Row Level Security
anon / authenticated (roles)
Two standing "badges" Supabase issues to anything that connects. anon is the badge given to any caller holding the public read-only key (today: the old dashboard, and anyone who has that key). authenticated is the badge given to someone who has actually logged in. RLS policies are written per badge — "authenticated users can read this table, anon cannot."

Backend, APIs & Auth

API / Endpoint
The dashboard's server (FastAPI, written in Python) exposes named "doors" that the browser can knock on to ask for data or request a change — e.g. POST /overrides, GET /triage. Each door (endpoint) has a fixed contract: what you're allowed to send it, and what it sends back. The whole set of doors is the API. Primary source: MDN — HTTP Overview
GET vs POST
Two of the "verbs" a request to an endpoint can use. GET means "just show me something," and is assumed safe to repeat with no side effects. POST means "do something" (create, change, delete) and is not safe to blindly repeat.
Router / Service (layers)
CMBSDOCS's backend code is organized in layers: a router file just defines "this URL exists, here's roughly what it does," and hands the real work to a service function, which is where the actual logic lives (talk to the database, apply business rules). This separation means the same logic (e.g. reclassifying a 15G filing) can be triggered from two different front doors — the dashboard and Slack — by both routers calling the same one service function, instead of two copies of the logic drifting apart.
JWT (JSON Web Token) / Auth
A signed, tamper-proof digital "ID card" issued when someone logs in, attached to every subsequent request so the server can verify who's asking without re-checking a password every time. Auth (authentication) is the umbrella term for "proving who you are"; authorization is the separate question of "what are you allowed to do now that we know who you are" (that's what RLS and the "must be logged in" endpoint guards handle). Primary source: Supabase Docs — JWTs
Actor / Actor attribution
Recording exactly who (which human, or "the system" for an automated process) made a given change, attached to that change permanently. Before this plan, most database changes had no such record — you could see that a deal was reclassified, but not who reclassified it or through which surface (dashboard vs. Slack).
CORS
A browser-enforced safety rule that blocks a webpage from one address from freely calling an API hosted at a different address, unless that API explicitly allows it. Adding the new dashboard's web address to the backend's CORS allow-list is a small, easy-to-forget step whenever a new frontend is stood up.

Frontend (the part you click on)

SPA (Single-Page Application)
A website that loads once, then swaps its content in and out in the browser using JavaScript, instead of asking the server for a whole new page on every click. The new dashboard is an SPA, built with React.
React
The JavaScript library used to build the dashboard's screens out of reusable, composable pieces called components (a table row, a filter dropdown, a whole page).
TanStack (Router / Query / Table / Virtual)
A family of four add-on tools the dashboard uses on top of React, each solving one specific, recurring problem:
  • Router — turns every filter/search choice into part of the page's URL, so a filtered view can be bookmarked or shared as a link.
  • Query — manages "ask the server for data, cache it, and update the screen instantly (optimistically) before the server even confirms the change."
  • Table — the engine behind sortable, filterable data grids.
  • Virtual — renders only the rows currently visible on screen for very long lists (e.g. thousands of log rows), so the browser doesn't have to draw all of them at once.
Primary source: TanStack Query docs, TanStack Router docs
Optimistic update
When you click an action (e.g. rename a deal), the screen updates immediately as if it already worked, while the real request is still in flight to the server. If the server later says "no, that failed," the screen quietly rolls back. This makes the app feel instant instead of laggy.
Stubbed (write)
A button or form that looks fully real and clickable, but instead of actually changing anything, just shows a small "not wired yet" message. Used deliberately in Phase 1 so the whole app's layout could be reviewed and approved before any real backend logic was built underneath it.

The Override System (the plan's centerpiece)

Override
A human's correction to a piece of data that a machine (an LLM extraction, a scraper, an automated matcher) originally produced. The core design rule of this whole plan: a human override must never be silently erased by the machine running again later.
Ledger (the field_overrides table)
One single table that records every override, ever made, on any field, on any table, as its own permanent row — instead of adding a separate "_override" column to every table for every field that might need one. It records the old (machine) value, the new (human) value, who made the change, and when. See the Phase 4 lesson for the full design.
Write-through
When a human overrides a value, the change is applied in two places at the same time, in the same database transaction: (1) a new row is added to the ledger as the permanent history, and (2) the actual base column (e.g. deals.deal_name) is updated to the new value. Write-through exists so that every other piece of code that reads deals.deal_name directly — a matcher, a Slack message, a list view — sees the corrected value automatically, without needing to know the override system exists.
Reassert (trigger)
The safety net for write-through: if some machine process updates a field that has an active override anyway (either because its code wasn't updated to skip overridden fields, or it's an external system CMBSDOCS doesn't control), a database trigger catches the incoming change before it's saved and puts the human's overridden value back in its place — while quietly recording what the machine tried to write as latest_machine_value, so you can still see "the model now disagrees with the human, here's what it currently thinks."
Guard (the Python-side guard)
The first, preferred line of defense: before a machine writer (an extractor, a matcher) saves its results, a shared helper checks which fields already have an active override and skips writing those specific fields. The reassert trigger is the backstop for when this guard is missing or bypassed — notably, one upstream system (an external "sec-chunker") can't be given this Python guard at all, which is exactly why the database-level trigger backstop is mandatory, not just a nice-to-have.
Cascade
When overriding one field has to trigger consistent follow-on changes elsewhere — e.g. reclassifying a deal's type has to also update its monitoring flags and reconcile its monitoring tasks, or the deal ends up in an inconsistent state (e.g. still being watched under the wrong asset-class rules).
Provenance (chip)
A small UI marker next to any data field showing where its current value came from: the LLM (with a confidence %), a human (locked), a scraper, or an external system. Clicking it shows the full override history.
Confirm / Dismiss (review states)
Two special kinds of ledger entries that aren't corrections: Confirm means a human looked at the machine's answer and endorsed it as-is (still protects it from being silently changed later). Dismiss means a human decided the whole record is irrelevant (e.g. "this filing isn't actually a CMBS deal"), with a required explanation.
Audit log
A separate, older table that automatically records every raw database change (old value → new value) via a trigger, regardless of whether it went through the override system. The ledger tracks the meaning of a change ("this was a human correction"); the audit log tracks the mechanical fact of a change ("this row's this column changed at this time").

Project & workflow terms

Branch (git branch)
An isolated copy of the codebase's history where changes can be made and tested without touching the shared, working version until they're ready and reviewed. The plan's progress log mentions branch names like dashboard-phase4 — that's "the isolated workspace where Phase 4's changes were built and tested."
Worktree
A lightweight second working copy of the same repository, used so a branch's changes can be tested on disk without disturbing whatever else is checked out.
Recon (live-DB recon)
Phase 2's work: before writing any code, actually go look at the real, live database to verify assumptions (What type is this ID column? Does this table already have RLS? Is this column really missing?) instead of guessing from documentation that might be stale. Several "corrections" in later phases exist specifically because recon caught a wrong assumption early.
E2E (End-to-end) test / verification
Actually running the whole real flow — not just checking the code compiles, but doing the actual action (override a value, then have the machine try to overwrite it, then confirm the override survived) and observing the real result.
Worker / Queue (RQ, Redis)
Some jobs (scraping a website, running an LLM extraction) take too long to do while a user waits, so they're placed on a queue (built on Redis, a fast in-memory data store) and picked up by background worker processes running separately from the main web server.
